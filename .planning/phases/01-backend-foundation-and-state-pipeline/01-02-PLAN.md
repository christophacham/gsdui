  <behavior>
    **Frontmatter extractor (src/parser/frontmatter.rs):**
    - Input: "---\nkey: value\n---\nbody" -> Output: Document { metadata: T, content: "body" }
    - Input: "no frontmatter" -> Output: Err(ParseError::NoFrontmatter)
    - Input: "---\nunclosed" -> Output: Err(ParseError::UnclosedFrontmatter)
    - Input: "---\ninvalid: [yaml\n---\n" -> Output: Err(ParseError::YamlError(...))
    - Input with leading whitespace/BOM handled correctly

    **STATE.md parser (src/parser/state_md.rs):**
    - Parses frontmatter: gsd_state_version, milestone, milestone_name, status, stopped_at, last_updated, last_activity, progress (total_phases, completed_phases, total_plans, completed_plans, percent)
    - Parses body sections via regex: Current Position (phase number, plan number), Performance Metrics (velocity stats)
    - Returns StateMdData struct with all fields as Option<T> for resilience

    **ROADMAP.md parser (src/parser/roadmap.rs):**
    - Extracts `### Phase N: Name` headings (supports decimal: 2.1, 2.2)
    - Extracts **Goal**, **Depends on**, **Requirements**, **Plans** count
    - Extracts plan checklist items with completion status
    - Extracts progress table (phase completion counts)
    - Returns Vec<PhaseEntry> sorted by phase number

    **PLAN.md parser (src/parser/plan.rs):**
    - Parses frontmatter: phase, plan, type, wave, depends_on (array), files_modified (array), autonomous, requirements (array), must_haves (object)
    - Returns PlanMdData with all fields

    **SUMMARY.md parser (src/parser/summary.rs):**
    - Parses frontmatter: phase, plan, duration, completed (datetime), key-files (array), requirements-completed (array), provides, affects
    - Parses body "## Task Commits" section: extracts numbered list items with backtick commit hashes, commit messages, and commit types (feat/fix/test/refactor/docs)
    - Returns SummaryMdData with commits as Vec<CommitRecord>

    **VERIFICATION.md parser (src/parser/verification.rs):**
    - Parses frontmatter: phase, status (passed/gaps_found/human_needed), score ("N/M" string)
    - Returns VerificationMdData

    **config.json parser (src/parser/config_json.rs):**
    - Parses full config.json structure: mode, granularity, workflow settings, planning settings, parallelization, gates, safety
    - Uses serde_json::Value for flexibility with unknown fields
    - Returns GsdConfig struct

    **agent-history.json parser (src/parser/agent_history.rs):**
    - Parses array of agent session records
    - Each record: agent_id, agent_type, phase, plan, started_at, ended_at
    - Uses flexible parsing (serde_json::Value fallback for unknown fields per user's strict mode)
    - Returns Vec<AgentSession>

    **Stage derivation (src/parser/stage.rs):**
    - Input: list of filenames in a phase directory + plan count from ROADMAP
    - No phase directory -> Planned
    - CONTEXT.md exists -> Discussed
    - RESEARCH.md exists -> Researched
    - PLAN.md exists -> Planned (execution-ready = planned_ready)
    - Some SUMMARY.md exist -> Executing
    - All SUMMARY.md exist (count matches plan_count) -> Executed
    - VERIFICATION.md exists -> Verified
    - Priority: later stages override earlier (Verified > Executed > Executing > ...)
  </behavior>
  <implementation>

    All parsers return Result<T, ParseError> where ParseError is a unified error enum with variants per failure mode. Per user decision: strict mode -- unknown fields generate warnings, not silent acceptance.

    Parser output types are owned structs (not DB row references). The mapping from parser output to DB rows happens in the watcher/pipeline layer (Plan 01-03), not here.
  </implementation>
