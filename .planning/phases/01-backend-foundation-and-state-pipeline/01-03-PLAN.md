<objective>
Implement the file watcher with custom per-file debouncing, the parse pipeline that connects file change events to the correct parser and writes results to the database, startup reconciliation for offline changes, and periodic data retention pruning.

Purpose: This is the data ingestion engine. Without it, the database never gets populated and the WebSocket has nothing to broadcast. The watcher + pipeline transforms filesystem events into structured state that the real-time layer consumes. Startup reconciliation ensures the daemon recovers from offline periods, and retention pruning keeps the database from growing unbounded.

Output: A running file watcher that monitors registered projects' .planning/ directories, debounces changes, dispatches to parsers, persists to SQLite, handles errors gracefully per user's strict mode decision, reconciles on startup, and prunes expired data.
</objective>

    1. Write src/watcher/pipeline.rs:
       - `pub async fn run_pipeline(mut event_rx: mpsc::Receiver<Vec<DebouncedEvent>>, db: SqlitePool, broadcast_tx: broadcast::Sender<StateUpdate>)`:
         - Receives batched debounced events
         - For each event, determines which parser to call based on filename:
           - "STATE.md" -> parser::parse_state_md
           - "ROADMAP.md" -> parser::parse_roadmap
           - Matches `{phase}-{plan}-PLAN.md` -> parser::parse_plan_md
           - Matches `{phase}-{plan}-SUMMARY.md` -> parser::parse_summary_md
           - Matches `{phase}-VERIFICATION.md` -> parser::parse_verification_md
           - Matches `{phase}-CONTEXT.md` or `{phase}-RESEARCH.md` -> stage derivation only (no content parsing needed, presence is what matters)
           - "config.json" -> parser::parse_config_json
           - "agent-history.json" -> parser::parse_agent_history
         - Reads file content via tokio::fs::read_to_string
         - Calls appropriate parser
         - On success: maps parser output to DB row types, calls db::schema upsert/insert functions
         - On parse error: records in parse_errors table per user decision (strict mode), retains last-known-good state (do NOT update DB row with partial/empty data)
         - After DB write: sends StateUpdate on broadcast channel (for Plan 01-04 to consume)
         - After file change in phase directory: re-derive stage via parser::derive_stage, update phase_state.stage in DB

           - Run the appropriate parser
           - Upsert results into database (upsert semantics ensure offline changes overwrite stale DB rows)

         - Clear resolved parse_errors for this project before re-parsing (old errors from files that no longer exist get cleaned up)

         a. Check if project path still exists on disk. If not: mark project status as "offline" in DB, log warning, skip to next project.
         b. If path exists: call pipeline::bootstrap_project(project_id, path, &db, &broadcast_tx) to trigger a full re-parse of all .planning/ files. This reconciles any changes made while the daemon was offline (files added, modified, or deleted).
         c. After successful bootstrap: add file watcher for the project via FileWatcher::watch_project
       - This ensures the DB is fully up-to-date before the daemon starts accepting WebSocket connections

    6. Write src/watcher/retention.rs -- Periodic data retention pruning:
       - `pub async fn run_retention_pruning(db: SqlitePool, cancel: CancellationToken)`:
         - Runs in a loop with configurable interval (default: every 24 hours). Use tokio::select! with cancel token for graceful shutdown.
         - On each tick:
           a. Query all projects and their retention_days (from projects table, default 180)
           b. For each project, calculate the cutoff timestamp: now - retention_days
           c. Hard-delete from SQLite all rows older than the cutoff in these tables:
              - execution_runs (and cascading to commits) where completed_at < cutoff
              - agent_sessions where ended_at < cutoff
              - parse_errors where occurred_at < cutoff AND resolved_at IS NOT NULL (only prune resolved errors)
           d. Do NOT delete phase_state or plan_state rows (these represent current state, not historical)
           e. Do NOT delete verification_results (lightweight, useful for historical trend)
           f. Log summary: "Retention pruning: deleted {N} execution runs, {M} agent sessions, {P} resolved parse errors for project {name}"

         - prune_resolved_errors(pool, project_id, cutoff) -> Result<u64>
